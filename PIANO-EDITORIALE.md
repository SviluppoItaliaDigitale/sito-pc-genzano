# Piano Editoriale — Sito Protezione Civile Genzano di Roma

**Obiettivo**: tendere a **un articolo al giorno** (circa 300-365 l'anno), con un minimo sostenibile di **3-4 articoli a settimana** nei periodi di minore attività.
**Strategia**: non produrre tutto da zero. **Rielaborare fonti ufficiali**, adattandole al territorio di Genzano e dei Castelli Romani, e ruotare su una biblioteca ampia di contenuti evergreen quando non ci sono notizie.

## 4. Trasformare una fonte in un articolo

### Passo 1 — Selezione
Consultare le fonti della sezione 1. Una notizia è pubblicabile se ricorre almeno una di queste condizioni:
- Riguarda il Lazio o i Castelli Romani
- È un aggiornamento su normativa o procedure
- Introduce un concetto utile ai cittadini
- È un'occasione per parlare di prevenzione

### Passo 2 — Rielaborazione per Genzano
- Riscrivere in linguaggio PA (AGID): frasi brevi, voce attiva, niente burocratese
- Contestualizzare al territorio ("Per Genzano e i Castelli Romani...")
- Aggiungere consigli pratici azionabili
- Citare la fonte con link (mai copiare interi paragrafi)
- Ricordare il 112 quando il tema è di emergenza
- Inserire rimandi interni al sito (pagine rischi, allerte, giochi didattici)

### Passo 3 — Pubblicazione
```bash
hugo new comunicazioni/AAAA-MM-GG-titolo-articolo.md
```
oppure chiedere a Claude Code di redigerlo. L'articolo deve rispettare:
- Frontmatter completo (`archetypes/comunicazioni.md`)
- Data in formato `AAAA-MM-GG`
- Badge coerente con il tipo di contenuto
- Immagine con fascia blu (`MANUALE-SITO.md` Parte 3) o fallback `notizia-default.svg`
- Alt text significativo
- Link interni relativi (`/rischi-prevenzione/`, non percorsi assoluti)

### Esempio
**Fonte**: il Centro Funzionale Regionale Lazio emette allerta gialla per temporali.
**Articolo**: "Allerta gialla per temporali: cosa fare a Genzano" (badge: Allerta, priorità: urgente, scadenza: 48 ore).

| Periodo | Articoli/settimana | Articoli/mese | Temi prevalenti |
|---|---|---|---|
| Gen-Feb | 5-7 | 20-30 | Evergreen, bilanci, rischio freddo, piano familiare |

| Mag-Giu | 7 | 28-30 | Preparazione AIB, scuole, ondate calore, eventi locali |
| Lug-Ago | 6-7 | 25-30 | AIB, caldo, laghi, turismo, Ferragosto |
| Set-Ott | 7 | 28-31 | Io Non Rischio, autunno, scuole, anniversari |
| Nov-Dic | 6-7 | 25-30 | Piogge, freddo, volontariato, bilancio, Capodanno |

**Obiettivo annuale**: 300-365 articoli. **Minimo sostenibile**: 180/anno (15/mese).
Nei giorni senza notizie, pescare dalla biblioteca evergreen (sezione 3).

## 6. Ritmo operativo settimanale

- **Lunedì mattina (15 min)**: giro delle fonti calde (DPC, INGV, Regione, meteo). Se trovi qualcosa, pianifica 1-2 articoli.
- **Lunedì-venerdì (20 min/giorno)**: un articolo al giorno pescando da fonti della settimana o dalla biblioteca evergreen.
- **Sabato (30 min)**: articolo di approfondimento (memoria storica, territorio, formazione).
- **Domenica**: nessuna pubblicazione salvo emergenze in corso.

In periodo di emergenza attiva (modalità `emergenza.json: attiva:true`), sospendere il calendario evergreen e concentrarsi solo su aggiornamenti operativi.

## 7. Rotazione dei badge

Per un sito equilibrato, nella media settimanale evitare di concentrarsi su un solo badge:

| Badge | Quota target settimanale |
|---|---|
| Informazione / Prevenzione | 40% (3 articoli/settimana) |
| Comunicazione / Aggiornamento | 20% (1-2 articoli) |
| Volontariato / Formazione | 20% (1-2 articoli) |
| Allerta / Avviso / Emergenza | variabile, dipende da eventi |
| Attività / Evento / Esercitazione | 10-15% (1 articolo) |
| Radiocomunicazioni | 1-2 articoli/mese |

## 8. Controllo qualità prima di pubblicare

Checklist rapida (vedi `MANUALE-SITO.md` per quella estesa):
- [ ] Frontmatter completo e data in formato `AAAA-MM-GG`
- [ ] Linguaggio AGID: frasi brevi, voce attiva, niente burocratese
- [ ] Numeri di emergenza corretti: nel Lazio **solo 112** per il cittadino
- [ ] Fonti citate con link ufficiale
- [ ] Immagine copertina con fascia blu o fallback `notizia-default.svg`
- [ ] Foto evento inserite con shortcode `{{< foto >}}` (non markdown diretto), con nome file diverso dallo slug per non essere sovrascritte da `genera-cover.py`
- [ ] Alt text significativo su tutte le immagini (mai "Immagine di...")
- [ ] Allegati PDF con campo `dimensione` (es. "120 KB") per WCAG 3.3.5 Help
- [ ] Link interni testati (nessun 404)
- [ ] Build Hugo pulito prima del push
